# iOS 埋点 SDK 实现计划

## [ ] 任务 1: 定义事件模型 (Event.swift)
- **优先级**: P0
- **依赖**: 无
- **描述**:
  - 创建 `Event` 结构体，包含 id、name、params 和 timestamp 字段
  - 创建 `AnyCodable` 结构体，解决 Any 类型不能 Codable 的问题
- **成功标准**:
  - Event 结构体可以正确初始化和序列化
  - AnyCodable 可以处理任意类型的值
- **测试要求**:
  - `programmatic` TR-1.1: 验证 Event 实例可以被正确创建和打印
  - `programmatic` TR-1.2: 验证 AnyCodable 可以处理不同类型的值
- **备注**: 这是整个 SDK 的基础数据模型

## [ ] 任务 2: 实现事件队列 (EventQueue.swift)
- **优先级**: P0
- **依赖**: 任务 1
- **描述**:
  - 创建 `EventQueue` 类，管理事件的内存队列
  - 实现线程安全的入队操作
  - 实现批量上报逻辑（达到20个事件时触发）
- **成功标准**:
  - 事件可以安全地加入队列
  - 达到阈值时自动触发批量上报
- **测试要求**:
  - `programmatic` TR-2.1: 验证多线程环境下事件入队的安全性
  - `programmatic` TR-2.2: 验证达到20个事件时触发flush
- **备注**: 线程安全使用 NSLock 实现

## [ ] 任务 3: 实现上传模块 (UploadManager.swift)
- **优先级**: P0
- **依赖**: 任务 1
- **描述**:
  - 创建 `UploadManager` 类，负责事件的网络上传
  - 实现异步上传逻辑
  - 模拟网络请求（后续可扩展为实际网络请求）
- **成功标准**:
  - 事件可以被异步上传
  - 上传过程不阻塞主线程
- **测试要求**:
  - `programmatic` TR-3.1: 验证上传方法被调用且参数正确
  - `programmatic` TR-3.2: 验证上传在后台线程执行
- **备注**: 后续可扩展重试机制、压缩和 protobuf 支持

## [ ] 任务 4: 实现统一埋点入口 (Analytics.swift)
- **优先级**: P0
- **依赖**: 任务 1, 任务 2
- **描述**:
  - 创建 `Analytics` 类，作为对外统一入口
  - 实现 `track` 方法，创建事件并加入队列
  - 添加常见埋点方法，包括：
    - 页面相关：页面显示、页面隐藏、页面停留时长
    - 用户交互：点击事件、滑动事件、长按事件
    - 网络相关：网络请求开始、网络请求结束、网络错误
    - 应用生命周期：应用启动、应用退出、应用进入后台、应用回到前台
    - 性能相关：启动时间、内存使用、CPU 使用率
    - 错误相关：崩溃、异常、错误日志
- **成功标准**:
  - 业务层可以通过 `Analytics.shared.track()` 方法埋点
  - 调用 `track` 后事件正确进入队列
  - 所有常见埋点类型都有对应的方法
- **测试要求**:
  - `programmatic` TR-4.1: 验证 `track` 方法可以创建并入队事件
  - `programmatic` TR-4.2: 验证所有常见埋点方法可以正常调用
  - `human-judgement` TR-4.3: 验证 API 调用方式简洁明了
- **备注**: 这是业务层使用的主要接口

## [ ] 任务 5: 实现自动页面曝光 (UIViewController+Track.swift)
- **优先级**: P1
- **依赖**: 任务 4
- **描述**:
  - 创建 `UIViewController` 扩展，使用 Method Swizzling 实现自动埋点
  - 在 `viewDidAppear` 时自动上报页面显示事件
- **成功标准**:
  - 所有 UIViewController 实例在显示时自动上报页面事件
  - 不影响原有功能
- **测试要求**:
  - `programmatic` TR-5.1: 验证视图控制器显示时自动触发埋点
  - `programmatic` TR-5.2: 验证原有 `viewDidAppear` 逻辑正常执行
- **备注**: 需要在 AppDelegate 中调用 `UIViewController.swizzle()`

## [ ] 任务 6: 实现列表曝光管理 (ExposureManager.swift)
- **优先级**: P1
- **依赖**: 任务 4
- **描述**:
  - 创建 `ExposureManager` 类，处理列表（UITableView）的曝光事件
  - 实现可见性判断逻辑（曝光比例 > 0.5）
  - 实现曝光去重逻辑
- **成功标准**:
  - 列表单元格在可见比例超过50%时触发曝光事件
  - 每个单元格只触发一次曝光事件
- **测试要求**:
  - `programmatic` TR-6.1: 验证可见比例计算正确
  - `programmatic` TR-6.2: 验证曝光事件去重
- **备注**: 需要在 VC 的 `scrollViewDidScroll` 方法中调用

## [ ] 任务 7: 实现后台强制上报
- **优先级**: P1
- **依赖**: 任务 2
- **描述**:
  - 在 AppDelegate 的 `applicationDidEnterBackground` 方法中调用 `EventQueue.shared.flush()`
  - 确保应用进入后台时所有待上报事件都被处理
- **成功标准**:
  - 应用进入后台时触发事件队列的 flush 操作
- **测试要求**:
  - `programmatic` TR-7.1: 验证应用进入后台时调用 flush 方法
- **备注**: 确保数据不丢失

## [ ] 任务 8: 实现磁盘缓存 (DiskStorage.swift)
- **优先级**: P2
- **依赖**: 任务 1
- **描述**:

  - 应用启动时恢复未上传的事件
- **成功标准**:
  - 事件可以被持久化到磁盘

- **测试要求**:

  - `programmatic` TR-8.2: 验证应用重启后事件恢复
- **备注**: 防止应用崩溃导致数据丢失
